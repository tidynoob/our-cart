    - initialize() sets isLoading: false after callback → FAIL

    - signInWithGoogle() sets error on failure → FAIL

    GREEN phase — write authStore.ts such that all 9 tests pass.

    Interface:
    - export interface AuthState: user (User | null), session (Session | null), isLoading (boolean), error (string | null), initialize (() => () => void), signInWithGoogle (() => Promise<void>), signOut (() => Promise<void>)
    - export const useAuthStore = create<AuthState>()((set) => ({ ... }))
    - Initial state: user: null, session: null, isLoading: true, error: null
    - initialize(): calls supabase.auth.onAuthStateChange with a SYNCHRONOUS callback (no async keyword on the callback), returns () => subscription.unsubscribe()
    - onAuthStateChange callback: set({ user: session?.user ?? null, session, isLoading: false, error: null }), then if session?.access_token call supabase.realtime.setAuth(session.access_token) else supabase.realtime.setAuth(null)
    - signInWithGoogle(): set({ error: null }), call supabase.auth.signInWithOAuth({ provider: 'google', options: { redirectTo: window.location.origin } }), if error set({ error: error.message })
    - signOut(): call supabase.auth.signOut(), if error set({ error: error.message })
  </behavior>
  <implementation>
    Follow the exact create<AuthState>()((set) => ({...})) pattern from itemsStore.ts. Import create from 'zustand', import type User and Session from '@supabase/supabase-js', import supabase from '@/lib/supabase'.

    The onAuthStateChange callback MUST NOT have the async keyword — this is a locked decision from STATE.md. The callback signature is: (event: AuthChangeEvent, session: Session | null) => void. Import AuthChangeEvent from '@supabase/supabase-js'.

    The realtime.setAuth call goes inside the synchronous callback body. If async work were ever needed from an auth event, it should be dispatched as a non-awaited call: asyncFn() — never await inside the callback.

    For signInWithGoogle, use window.location.origin as redirectTo. This works for both localhost:5173 and the Vercel production domain because the Supabase allowlist validates the final redirectTo value. No hardcoded URLs.

    Use the path alias @/lib/supabase (not relative imports) — all project files use this convention per PATTERNS.md.
  </implementation>
</feature>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser localStorage → authStore | Session tokens written by Supabase Auth client; read by onAuthStateChange on page load |
| authStore → Supabase Realtime | JWT passed via setAuth(); old channel sessions cannot be hijacked because setAuth() updates per-connection |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-03 | Information Disclosure | Session token in localStorage | Accept | Supabase Auth default behavior; tokens are short-lived (1 hour) with auto-refresh; no PII beyond email/name stored by app |
| T-06-04 | Spoofing | Async onAuthStateChange callback (missed events) | Mitigate | Callback is synchronous per D-08/STATE.md locked decision; any async work dispatched without await to prevent missed SIGNED_OUT/TOKEN_REFRESHED events |
| T-06-05 | Elevation of Privilege | Realtime channel using stale anonymous JWT after sign-in | Mitigate | supabase.realtime.setAuth(session.access_token) called in every onAuthStateChange callback invocation (per STATE.md locked decision) |
| T-06-SC | Tampering | npm/pip/cargo installs | Accept | No new package installs — authStore uses zustand and @supabase/supabase-js already installed (RESEARCH.md confirms no new dependencies) |
</threat_model>
