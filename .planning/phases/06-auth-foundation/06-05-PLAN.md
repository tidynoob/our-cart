    Write supabase/migrations/items_auth.sql per RESEARCH.md Pattern 7:

    Step 1 — Add nullable user_id column with DEFAULT:
    ALTER TABLE items ADD COLUMN user_id uuid REFERENCES auth.users(id) ON DELETE SET NULL DEFAULT (select auth.uid());

    The DEFAULT (select auth.uid()) ensures new items inserted by authenticated users automatically get their user_id set at the database level, even if the application forgets to include it in the INSERT payload (per RESEARCH.md Open Question 2 recommendation).

    Step 2 — Drop ALL existing items policies using DROP POLICY IF EXISTS for every plausible policy name from v1.0 and any that might have been added in later phases. Use IF EXISTS on all drops to ensure idempotency:
    DROP POLICY IF EXISTS "anon_select_items" ON items;
    DROP POLICY IF EXISTS "anon_insert_items" ON items;
    DROP POLICY IF EXISTS "anon_update_items" ON items;
    DROP POLICY IF EXISTS "anon_delete_items" ON items;

    Step 3 — Create four new policies (SELECT, INSERT, UPDATE, DELETE) using the pattern: user_id IS NULL OR (select auth.uid()) = user_id. Grant to both anon and authenticated roles. INSERT and UPDATE policies need WITH CHECK in addition to USING. Use policy names: "items_select", "items_insert", "items_update", "items_delete".

    Do NOT run ENABLE ROW LEVEL SECURITY or FORCE ROW LEVEL SECURITY — already enabled from Phase 1.

    Write supabase/migrations/lists_auth.sql per RESEARCH.md Pattern 8:

    Step 1 — Add nullable owner_id column (no DEFAULT — lists are created in application code, which will pass owner_id explicitly in Phase 7):
    ALTER TABLE lists ADD COLUMN owner_id uuid REFERENCES auth.users(id) ON DELETE SET NULL;

    Step 2 — Drop existing lists policies:
    DROP POLICY IF EXISTS "anon_select_lists" ON lists;
    DROP POLICY IF EXISTS "anon_insert_lists" ON lists;
    DROP POLICY IF EXISTS "anon_update_lists" ON lists;
    DROP POLICY IF EXISTS "anon_delete_lists" ON lists;

    Step 3 — Create new policies using owner_id IS NULL OR (select auth.uid()) = owner_id pattern. Policy names: "lists_select", "lists_insert", "lists_update", "lists_delete". Grant to anon and authenticated. Note: lists_insert and lists_update need both USING and WITH CHECK.

    Add a comment header to each SQL file: -- Phase 6: Auth Foundation — {table} auth scaffolding (D-09/D-10 or D-11)

    Do not add a lists_delete policy at this stage — Phase 7 will handle list deletion with ownership checks. A lists_delete policy is deferred to Phase 7 to avoid accidental deletion of lists without proper ownership verification. Omission is intentional.

    Actually — add lists_delete policy now for completeness. Policy: owner_id IS NULL OR (select auth.uid()) = owner_id. This preserves the same permissive pattern as other policies while Phase 7 can tighten it.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| RLS policies → database queries | Postgres enforces RLS on every query; policy correctness is the security boundary |
| user_id DEFAULT → auth.uid() | DEFAULT function runs in database context; auth.uid() returns null for anonymous requests |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-11 | Elevation of Privilege | Anonymous user accessing another user's items via direct API call | Mitigate | items RLS policy: user_id IS NULL OR (select auth.uid()) = user_id. Authenticated users can only access their own items (or items without user_id). Legacy anonymous items accessible to all — by design for backward compatibility (D-09). |
| T-06-12 | Elevation of Privilege | Anonymous user accessing another user's lists | Mitigate | lists RLS policy: same pattern. Lists without owner_id accessible to all — intentional for Phase 6; Phase 7 will tighten with share-code + membership checks. |
| T-06-13 | Denial of Service | DROP POLICY mid-migration leaves table unprotected | Mitigate | DROP POLICY IF EXISTS + CREATE POLICY are in the same SQL script executed as a single transaction in Supabase SQL Editor. If CREATE POLICY fails, the entire script fails and old policies remain. |
| T-06-14 | Tampering | user_id DEFAULT (select auth.uid()) set to null for anonymous inserts | Accept | Correct behavior — anonymous inserts produce user_id = null (no authenticated session). Policy allows null user_id items. New authenticated items will have user_id populated from session. |
| T-06-SC | Tampering | npm/pip/cargo installs | Accept | No new package installs in this plan. SQL-only migration. |
</threat_model>
