<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| URL params → DB query | `code` from `useParams()` goes directly into `.eq('share_code', code)` — must be validated before reaching the query |
| Supabase error → UI | Raw Supabase error objects must never render to the user |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | Supabase error exposed in ListPage render | Mitigate | ListPage renders only generic `'List not found'` on error — never `error.message` from Supabase (V7). Verified in ListPage.test.tsx. |
| T-02-02 | Elevation of Privilege | Share code brute force via repeated GET /list/:code | Accept | 64^8 ≈ 281 trillion combinations; no rate limiting at this scale; acceptable for private 2-person use. |
| T-02-03 | Tampering | List name rendered via XSS | Mitigate | React JSX escapes all interpolated values; list name is rendered as text in `<h1>` with no `dangerouslySetInnerHTML`. |
| T-02-04 | Spoofing | Invalid share_code format sent to Supabase | Mitigate | Plan 03's JoinListForm validates the extracted code with regex `/^[A-Za-z0-9_-]{8}$/` before navigation (V5). ListPage guards against empty `code` via early `navigate('/')`. |
</threat_model>
