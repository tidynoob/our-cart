    Interface ListsState: lists: List[], loading: boolean, error: string | null, and 4 actions: fetchLists(userId: string): Promise<void>, createList(name: string, userId: string): Promise<string>, renameList(id: string, name: string): Promise<void>, deleteList(id: string): Promise<void>

    Note on createList return type: return the share_code so callers (CreateListForm, LandingPage) can navigate to /list/:shareCode after creation. Return the shareCode string (from the optimistic row) or the real row's share_code on success. Return empty string on error.

    fetchLists(userId):
    - set({ loading: true, error: null })
    - supabase.from('lists').select('id, name, share_code, owner_id, created_at').eq('owner_id', userId).order('created_at', { ascending: false })
    - On error: set({ error: 'Failed to load lists', loading: false })
    - On success: set({ lists: data ?? [], loading: false })

    createList(name, userId):
    - Trim name first; if empty after trim, set error 'Failed to create list' and return ''
    - const tempId = nanoid(), shareCode = nanoid(8) per D-01
    - Optimistic row: { id: tempId, name: name.trim(), share_code: shareCode, owner_id: userId, created_at: new Date().toISOString() }
    - set((state) => ({ lists: [optimisticList, ...state.lists] })) — prepend (newest first)
    - supabase.from('lists').insert({ name: name.trim(), share_code: shareCode, owner_id: userId }).select().single()
    - On error: remove optimistic row (filter by tempId), set error 'Failed to create list', return ''
    - On success (data): replace tempId row with real data row, return data.share_code

    renameList(id, name):
    - Trim name; if empty after trim, set error 'Failed to rename list', return
    - CRITICAL: const prev = get().lists.find((l) => l.id === id) — snapshot BEFORE set() (Pitfall 4)
    - If !prev return
    - set((state) => ({ lists: state.lists.map((l) => (l.id === id ? { ...l, name: name.trim() } : l)) }))
    - supabase.from('lists').update({ name: name.trim() }).eq('id', id)
    - On error: rollback to prev name, set error 'Failed to rename list'

    deleteList(id):
    - CRITICAL: const prev = get().lists.find((l) => l.id === id) — snapshot BEFORE set()
    - If !prev return
    - set((state) => ({ lists: state.lists.filter((l) => l.id !== id) }))
    - supabase.from('lists').delete().eq('id', id)
    - On error: re-insert prev — set((state) => ({ lists: [...state.lists, prev], error: 'Failed to delete list' }))

    Export: export const useListsStore = create<ListsState>()((set, get) => ({ ... }))

    Do NOT include: channel, syncStatus, subscribeToList, unsubscribe — per D-06 there is no realtime on lists table in Phase 7.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| store action → Supabase | User-controlled inputs (list name) cross from client into Supabase query. RLS enforces server-side authorization. |
| client → DB (owner_id) | The createList action sets owner_id = userId from authStore — this is the authorization claim. RLS backs it up server-side. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Information Disclosure | fetchLists | mitigate | Query scoped to .eq('owner_id', userId) per D-05 — only the authenticated user's lists are returned. RLS `lists_select` provides server-side enforcement as a backstop. |
| T-07-02 | Tampering | createList | mitigate | owner_id = userId (from authStore.user.id) set explicitly on insert per D-04, closing Phase 6 NULL tech debt. Supabase authed session prevents spoofing userId via RLS. |
| T-07-03 | Tampering | renameList / deleteList | mitigate | Mutations call .eq('id', listId). Existing RLS policies (lists_update WITH CHECK owner_id = auth.uid(), lists_delete USING owner_id = auth.uid() OR IS NULL) block cross-user mutations at DB layer. No new policies required. |
| T-07-04 | Tampering | list name input | mitigate | name.trim() + non-empty guard in createList and renameList before any Supabase call. No raw SQL concatenation — parameterized via Supabase JS client. |
| T-07-05 | Information Disclosure | error messages | mitigate | Store actions use generic strings ('Failed to load lists', 'Failed to create list', etc.) — raw Supabase error.message never exposed to state or UI. |
| T-07-SC | Tampering | npm installs | accept | No new packages — all dependencies verified in prior phases. No slopcheck gate needed. |
</threat_model>
