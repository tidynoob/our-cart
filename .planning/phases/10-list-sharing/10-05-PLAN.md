    - InvitePage.tsx — join flow page with spinner + redirect + invalid-invite state
    - /invite/:code route registered (inside ProtectedRoute, outside AppShell)

  </what-built>
  <how-to-verify>
    You need two different Google accounts and two browser sessions (e.g., normal window + incognito). Deploy or run the dev server first.

    **Setup:** In Browser A (owner), create a list and add 2-3 items. Open the list. The ShareBanner should appear with a "Copy link" button.

    ---

    **Scenario 1 — Basic invite + member sees items:**
    1. In Browser A, click "Copy link" in the ShareBanner.
    2. Confirm the copied URL contains "/invite/" (not "/list/").
    3. Open the copied URL in Browser B (partner, different Google account, not yet signed in).
    4. Browser B should redirect to login page (ProtectedRoute returnTo flow).
    5. Sign in with Browser B's Google account.
    6. Browser B should automatically navigate to /invite/:code, then to /list/:code.
    7. Browser B (partner) should see all items that Browser A (owner) added.
    PASS: Partner sees items. FAIL: Empty list or RLS error.

    ---

    **Scenario 2 — Non-member blocked:**
    1. Use a third account (or reset Browser B's session and use a different account that has NOT redeemed the invite).
    2. In the Supabase SQL Editor, run: `SELECT * FROM items WHERE list_id = '<the list id>';` while connected as the non-member account.
    3. Expected: 0 rows (RLS blocks).
    Alternatively: in Browser B after signing out and signing in with a non-member account, navigate directly to `/list/:code` (the direct list URL). Expected: list not found / empty.
    PASS: Non-member cannot see items. FAIL: Items visible.

    ---

    **Scenario 3 — Idempotent redeem:**
    1. With Browser B (already a member), open the same /invite/:code URL again.
    2. Should redirect to /list/:code without error.
    3. In Supabase Table Editor → list_members, confirm only ONE row exists for (list_id, Browser B's user_id).
    PASS: Single row, no error on re-redeem. FAIL: Error or duplicate row.

    ---

    **Scenario 4 — Owner self-redeem:**
    1. In Browser A (owner), open the /invite/:code URL for their own list.
    2. Should redirect to /list/:code without error.
    3. list_members table should have NO row for the owner (owner access via owner_id, not list_members).
    PASS: Redirects cleanly; no row added. FAIL: Error or "invalid invite" shown.

    ---

    **Scenario 5 — Realtime cross-user:**
    1. Both Browser A (owner) and Browser B (member) have /list/:code open simultaneously.
    2. In Browser A, add a new item.
    3. Within ~2 seconds, Browser B should show the new item without any refresh.
    4. In Browser B, check off an item.
    5. Browser A should see the checked state update within ~2 seconds.
    PASS: Real-time updates flow both ways. FAIL: Updates require page refresh.

    ---

    **Scenario 6 — Unknown invite code:**
    1. Navigate to `/invite/NOTEXIST` in any signed-in browser.
    2. Should show "This invite link is invalid or has expired." with a "Back to home" link.
    PASS: Invalid UI shown. FAIL: Error page or redirect loop.

    ---

    **Scenario 7 — Unauthenticated partner flow (full round-trip):**
    1. Open a fresh incognito window (not signed in).
    2. Navigate directly to `/invite/:code`.
    3. Should redirect to login page (/).
    4. Sign in with Google.
    5. After OAuth redirect, should automatically navigate back to /invite/:code.
    6. InvitePage spinner appears, then redirects to /list/:code.
    7. List loads with all items visible.
    PASS: Complete flow works without manual navigation. FAIL: Lands on home page or invite URL lost after OAuth.
