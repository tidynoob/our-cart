<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client store → Supabase | `clearChecked` sends a bulk DELETE scoped by list_id and checked=true; RLS ensures the list_id matches the authenticated list |
| User intent → bulk destructive action | Modal dialog acts as the intent gate; backdrop dismiss is disabled to prevent accidental loss |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-04 | Tampering | `clearChecked` Supabase DELETE | mitigate | Phase 2 DELETE RLS policy already scopes to list_id context — a client cannot delete items from another list. The query always passes `list_id` from the loaded list object, never from user input. |
| T-03-05 | Tampering | Client passes wrong listId to clearChecked | mitigate | `list.id` is fetched from Supabase at page load and is a UUID set by the server. The client never constructs this value; it is read-only from the fetch. RLS validates it server-side on DELETE regardless. |
| T-03-06 | Denial of Service | User repeatedly tapping "Clear Items" after rollback | accept | 2-user app; Supabase free tier has 200 concurrent connections and 2M messages/month. Rollback restores items, preventing a cascade of empty DELETE calls (early return guard when checkedItems.length === 0). |
| T-03-07 | Repudiation | No confirmation log of bulk delete | accept | This is a personal grocery list between two people with no audit requirements. The Supabase DELETE is permanent but the action is explicitly confirmed via modal. |
| T-03-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed this phase. All dependencies pre-existing. No slopcheck gate needed. |
</threat_model>
