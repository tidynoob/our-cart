## Color

All tokens are declared in src/index.css. This phase adds no new color tokens — it uses the existing
semantic token set.

| Role | Token | Light Mode Value | Usage |
|------|-------|-----------------|-------|
| Dominant (60%) | --background | oklch(1 0 0) — white | Page surface, row background |
| Secondary (30%) | --secondary / --muted | oklch(0.97 0 0) — near-white gray | Edit row bg (`bg-secondary/50`), dialog footer (`bg-muted/50`), hover/active row states |
| Accent (10%) | --primary | oklch(0.205 0 0) — near-black | Filled checkbox background (`data-[checked]:bg-primary`), checkbox border filled state |
| Destructive | --destructive | oklch(0.577 0.245 27.325) — red | "Clear Items" confirm button in modal (`variant="destructive"`) only |

Accent (`--primary`) reserved for: filled checkbox indicator background only. No other elements in
this phase use primary as a fill.

Semantic tokens used this phase and their roles:
- `--border` (oklch 0.922): row dividers, checkbox border unchecked state (`border-input` = same token)
- `--muted-foreground` (oklch 0.556): quantity label, unknown-attribution badge text
- `--primary-foreground` (oklch 0.985 — white): Check lucide icon inside filled checkbox
- `--destructive`: Clear Items button background in confirmation dialog

Checked-item dimming:
- `opacity-50` applied to the entire row div when `item.checked === true`.
- The `opacity-50` class is the exact Tailwind utility (maps to `opacity: 0.5`).
- Applied at the outermost display-mode div so it dims the badge, name, and quantity uniformly.
- The checkbox itself is EXCLUDED from the opacity wrapper — it must remain full-opacity so the
  checked state is visually clear. Implementation: apply opacity to a sibling wrapper containing
  badge + name + quantity, or use `opacity-50` on the row and accept the checkbox dimming (simpler,
  acceptable since the `data-[checked]` fill color remains visible through 50% opacity).
- Decision: apply `opacity-50` to the full row div. At 50% opacity the filled checkbox circle
  remains readable. If visual testing reveals the checkbox becomes ambiguous, move opacity to the
  right-side sub-div (badge + name + qty). Default: full-row `opacity-50`.
- Source: CONTEXT.md D-05.

## Component Contract: Checkbox

This section specifies the new `src/components/ui/checkbox.tsx` component for this phase.

### Visual Anatomy

```
[  44px tap target wrapper  ]
  [ 20px circle visual ]
    border: 1px solid --input (unchecked)
    fill:   --primary (checked, via data-[checked]:bg-primary)
    border: 1px solid --primary (checked)
    radius: rounded (circular — matches existing badge style)
    check:  lucide Check icon, h-3.5 w-3.5, text-primary-foreground
```

### States

| State | Checkbox Visual | Ring | Name Text | Row Opacity |
|-------|----------------|------|-----------|-------------|
| Unchecked (default) | Empty circle, `border-input bg-background` | None | Normal, no decoration | 1.0 (full) |
| Checked | Filled circle, `bg-primary border-primary` + Check icon | None | `line-through` | 0.5 |
| Focus-visible (keyboard) | Unchecked or checked ring | `outline-ring/50` (global base style from index.css) | Unchanged | Unchanged |
| Disabled | Not applicable this phase | — | — | — |

### Data Attribute (Base UI — critical)

Base UI uses `data-[checked]` presence attribute, NOT Radix's `data-[state="checked"]`.
Tailwind v4 variant syntax: `data-[checked]:bg-primary`, `data-[checked]:border-primary`.
Source: RESEARCH.md Pitfall 2, verified against node_modules/@base-ui/react CheckboxRoot.d.ts.

### Tap Target Enforcement

Outer wrapper div: `flex h-[44px] w-[44px] items-center justify-center`
Checkbox.Root inside: renders the 20px visual circle.
This ensures 44px tap target on mobile without relying on className prop-forwarding to the Base UI
Root span element.

### Gesture Isolation

The checkbox wrapper div must have `onClick={(e) => e.stopPropagation()}` and
`onKeyDown={(e) => e.stopPropagation()}` to prevent the row's `onClick={onTap}` (edit mode trigger)
from firing when the checkbox is tapped.
Source: RESEARCH.md Pattern 5, Pitfall 1.

## Component Contract: Clear Confirmation Dialog

Reuses the existing `Dialog`, `DialogContent`, `DialogHeader`, `DialogTitle`, `DialogFooter`
pattern from `src/components/ui/dialog.tsx`. Same controlled `open` + `onOpenChange` pattern
as `NamePromptDialog`.

| Property | Value |
|----------|-------|
| Trigger | `clearDialogOpen` state in ListPage, set to `true` on "Clear completed" button click |
| `showCloseButton` | `false` — no X button in top-right; actions are Keep Items and Clear Items only |
| `disablePointerDismissal` | `true` — prevent accidental backdrop dismiss on mobile (one-handed shopping). Source: RESEARCH.md Pitfall 5, A2. |
| Title | `Remove {N} checked item{N !== 1 ? 's' : ''}?` — singular/plural correct. |
| Description | None — the title is self-explanatory for a grocery list context. |
| Footer layout | `DialogFooter` default: `flex-col-reverse` on mobile, `flex-row justify-end` on sm+. Keep Items on left/bottom, Clear Items on right/top on mobile. |
| Keep Items button | `variant="outline"`, label `Keep Items`. `onClick={() => setClearDialogOpen(false)}`. Plain Button, not DialogClose primitive. |
| Clear Items button | `variant="destructive"`, label `Clear Items`. `onClick` closes dialog then calls `clearChecked`. |
| Button order in JSX | Keep Items first, Clear Items second. `flex-col-reverse` makes Clear Items appear on top on mobile — matches mobile destructive-action convention. |

## Interaction Contract

### Check Off (SHOP-01, SHOP-02)

1. User taps the 44px checkbox area on an unchecked item row.
2. `e.stopPropagation()` fires — row-body `onClick={onTap}` does NOT fire. Edit mode does NOT open.
3. `onCheckedChange` calls `onToggle(item.id)` which calls `itemsStore.toggleChecked(id)`.
4. Optimistic update: `item.checked` flips to `true` in the store immediately.
5. Row re-renders: checkbox fills (`data-[checked]:bg-primary`), name gets `line-through`, row gets `opacity-50`.
6. Supabase UPDATE runs async. On success: no-op. On error: rollback + error banner.

### Uncheck (SHOP-02)

Same as check-off. The checkbox is already filled. Tapping it calls `toggleChecked(id)` again.
Optimistic update flips `item.checked` to `false`. Visual state resets instantly.

### Clear Completed (SHOP-03, SHOP-04)

1. `checkedCount >= 1` → "Clear completed (N)" button is rendered.
2. User taps button → `setClearDialogOpen(true)`.
3. Dialog opens with `disablePointerDismissal={true}`. Backdrop tap does nothing.
4. User taps "Keep Items" → `setClearDialogOpen(false)`. No data change.
5. User taps "Clear Items" → `setClearDialogOpen(false)` first, then `clearChecked(list.id)`.
6. Optimistic update: all checked items removed from store immediately. Button disappears (count=0).
7. Supabase DELETE runs async. On success: no-op. On error: rollback restores items + error banner.

### Keyboard / Accessibility

- Checkbox: receives focus via Tab. Space or Enter toggles checked state (Base UI Checkbox.Root
  handles this natively via the hidden `<input type="checkbox">`).
- Row body: Tab-focusable (`tabIndex={0}`). Enter or Space opens edit mode (pre-existing behavior,
  unchanged).
- Dialog: Base UI Dialog traps focus inside the modal. Escape closes it (calls `onOpenChange(false)`),
  which sets `setClearDialogOpen(false)`. This is equivalent to Keep Items — no data change.
- Clear Items button: standard Button, Tab-focusable, Enter activates.
