## Phase Goal

**As a** shopper, **I want to** check off items as I add them to my cart, **so that** I can see at a glance what's already in the cart and what still needs to be grabbed.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client store → Supabase | `toggleChecked` sends an UPDATE; RLS scopes this to the list's items only |
| Client optimistic state → render | Optimistic flip happens before DB confirmation; desync possible if network fails |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | `toggleChecked` Supabase UPDATE | mitigate | Phase 2 RLS UPDATE policy already requires `list_id` match — a client cannot toggle items belonging to a different list. No new policy needed. |
| T-03-02 | Spoofing | Optimistic UI showing wrong checked state | accept | Phase 3 has no real-time sync (Phase 4). Both users see their own optimistic state until next load. Rollback corrects local state on error. Documented for Phase 4 resolution. |
| T-03-03 | Denial of Service | Rapid checkbox tapping sending many UPDATEs | accept | 2-user app; Supabase free tier allows 200 concurrent connections and 2M messages/month — far beyond any realistic tap rate. |
| T-03-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed this phase — @base-ui/react 1.5.0 already in package.json. No slopcheck gate needed. |
</threat_model>
