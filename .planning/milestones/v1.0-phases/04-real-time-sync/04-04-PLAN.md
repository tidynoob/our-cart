<objective>
Fix the diagnosed UAT gap: SyncStatus pill fails to update to 'Reconnecting...' when the network drops because detection relied solely on the Supabase WebSocket heartbeat (25-50s delay). REST mutations fail instantly on network loss but never updated syncStatus. No window 'offline' event listener existed for immediate detection.

Root cause (from 04-HUMAN-UAT.md): Three missing behaviors —
1. No 'offline' event listener in ListPage → slow detection via WS heartbeat only
2. 'online' handler called fetchItems but did not re-subscribe or set syncStatus → status stuck on prior state
3. Mutation error handlers had no awareness of navigator.onLine → errors shown but pill stayed green

Purpose: Make the SyncStatus pill accurately reflect network state so users understand when changes are not syncing. This is the sole UAT blocker for Phase 04.

Output:
- ListPage.tsx with 'offline' handler (sets syncStatus 'reconnecting') and updated 'online' handler (sets 'connecting' + re-subscribes)
- itemsStore.ts with !navigator.onLine guard in five mutation error paths
- Test coverage for all three new behaviors
</objective>

**ListPage.tsx changes — update the 'online'/'offline' section of the second useEffect (the `if (!list) return` block):**

Replace the existing `handleOnline` function and `window.addEventListener('online', handleOnline)` registration with three handlers:

1. `handleOffline`: calls `useItemsStore.getState().setSyncStatus('reconnecting')` — sets the pill immediately on network loss.

2. `handleOnline` (replace the existing one): calls `useItemsStore.getState().setSyncStatus('connecting')` then calls `useItemsStore.getState().subscribeToList(list.id)`. This replaces the bare `fetchItems` call — re-subscribing triggers the full SUBSCRIBED → 'live' → fetchItems flow internally, so manual fetchItems is not needed and would duplicate the fetch.

3. Register `window.addEventListener('offline', handleOffline)` alongside the existing `document.addEventListener('visibilitychange', handleVisibility)` and `window.addEventListener('online', handleOnline)`.

4. Add `window.removeEventListener('offline', handleOffline)` to the useEffect cleanup return function alongside the existing removals.

Note: `useItemsStore.getState()` is already used for `handleVisibility` and `handleOnline` in this file — follow the same pattern to avoid stale closure issues (per D-07 comment in existing code).

**itemsStore.ts changes — add !navigator.onLine guard to five mutation error paths:**

In each mutation's `if (error)` block (addItem, updateItem, deleteItem, toggleChecked, clearChecked), add `syncStatus: !navigator.onLine ? 'reconnecting' : get().syncStatus` to the set() call alongside the existing `error:` and rollback fields.

The spread pattern for each is: include all existing rollback fields in the set() call, then add `syncStatus: !navigator.onLine ? 'reconnecting' : get().syncStatus`.

For `addItem` the set() call is the one that sets `error: 'Failed to add item'` — add syncStatus there.
For `updateItem` the set() is the one that sets `error: 'Failed to update item'` — add syncStatus there.
For `deleteItem` the set() is the one that sets `error: 'Failed to delete item'` — add syncStatus there.

For `clearChecked` the set() is the callback-form set() that sets `error: 'Failed to clear items'` — add `syncStatus: !navigator.onLine ? 'reconnecting' : get().syncStatus` inside the returned object.

Do NOT add a standalone `setSyncStatus` method to the store interface — the `set({ syncStatus: ... })` pattern already covers ListPage's needs via `useItemsStore.getState()` for the offline/online handlers.
