The UPDATE policy uses both USING and WITH CHECK clauses (both required for row-level UPDATE). The DELETE policy uses only USING (WITH CHECK is not applicable to DELETE). Both policies scope access via the list_id foreign key existing in the lists table, matching the exact security model of the existing SELECT and INSERT policies from Phase 1. Per D-08 and review concern #2 (RLS policy specificity).</what-built>
  <how-to-verify>
    1. Open your Supabase project dashboard -> SQL Editor
    2. Paste and run the following SQL:

    ALTER TABLE items ADD COLUMN added_by text;

    CREATE POLICY "anon_update_items" ON items FOR UPDATE TO anon
      USING (EXISTS (SELECT 1 FROM lists WHERE lists.id = items.list_id))
      WITH CHECK (EXISTS (SELECT 1 FROM lists WHERE lists.id = items.list_id));

    CREATE POLICY "anon_delete_items" ON items FOR DELETE TO anon
      USING (EXISTS (SELECT 1 FROM lists WHERE lists.id = items.list_id));

    3. Verify: In Table Editor, click on "items" table -> columns should now show "added_by" (text, nullable)
    4. Verify: In Authentication -> Policies -> items table should now show 4 policies: anon_select_items, anon_insert_items, anon_update_items, anon_delete_items
    5. Verify UPDATE policy has both USING and WITH CHECK. DELETE policy has only USING.
  </how-to-verify>

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Elevation of Privilege | items table UPDATE | mitigate | RLS policy anon_update_items with USING (list_id FK exists) AND WITH CHECK (list_id FK exists) — both clauses required for UPDATE |
| T-02-02 | Elevation of Privilege | items table DELETE | mitigate | RLS policy anon_delete_items with USING (list_id FK exists) — WITH CHECK not applicable to DELETE |

| T-02-SC | Tampering | npm/pip/cargo installs | mitigate | No new npm packages installed — shadcn CLI copies files using already-installed @base-ui/react (verified in Phase 1 Package Legitimacy Audit) |
